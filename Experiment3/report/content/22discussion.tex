\section{实验总结与讨论}

\subsection{实验过程回顾}

本实验围绕 SM4-CBC 在 RISC-V 平台上的实现与优化展开，依次完成了未优化版本、查表优化版本、多缓冲区并行加密、并行解密、Zksed 指令优化以及 bit-slice 常量时间实现的编码与测试。实验中同时保留了 x86 主机环境下的参考运行结果，用于调试验证和对照观察。

\subsection{RISC-V平台上的实验结论}

\subsubsection{查表优化是最稳定的性能优化}

从实验结果看，查表优化在 RISC-V 平台上是最有效、最稳定的优化手段。它不依赖专用硬件扩展，也不依赖平台特定指令，仅通过减少轮函数中重复的移位和异或操作，就在单流、多流和并行解密场景中都取得了稳定收益。
这一点具有很强的工程价值：当目标平台仅支持通用 RV64 指令、且软硬件栈较为基础时，查表优化仍然能够以较低实现复杂度带来明确的性能回报。

\subsubsection{编译器优化对RISC-V影响极大}
本实验最明显的现象之一是 RISC-V 平台对编译器优化高度敏感。未优化版本从 O0 到 O3 的提升超过 7 倍，说明在精简指令集平台上，循环展开、寄存器分配和函数内联对性能影响极其直接。
这意味着在 RISC-V 平台上部署密码算法时，代码本身的结构固然重要，但编译器优化级别往往同样决定了最终性能上限。对于实际部署而言，O2/O3 应作为默认配置。

\subsubsection{Zksed实现具备现实意义}
Zksed 指令优化版本在 QEMU-RISC-V 平台上能够正确运行，并在 O3 条件下达到与查表版本接近的吞吐量。虽然当前结果尚未显示其明显超越查表优化，但这并不意味着指令优化无效。
可能合理的理解是：在当前 QEMU 系统模拟环境中，专用密码指令的理论收益被模拟器翻译开销削弱了。换言之，本实验已经验证了该方向的可实现性，但其真实性能潜力仍需在支持更完整密码扩展的工具链、模拟器或真实硬件上进一步观察。

\subsubsection{Bit-Slice在当前环境中不适合作为性能方向的优化}

bit-slice 是本次实验中性能最低的方案。其吞吐量远低于查表版本和 Zksed 指令优化版本，说明在当前软硬件条件下，它并不适合作为高性能优化方案。
根本原因在于：bit-slice 的实现将原本一次 S 盒查表展开为大量布尔逻辑门操作，软件实现中的指令数急剧增加；同时在 QEMU 环境中，这些逻辑指令还要经过逐条翻译执行，导致执行代价进一步放大。因此，bit-slice 在本实验中的主要意义并不在于提高速度，而在于说明在 RISC-V 上实现常量时间 SM4 的方法是存在的。
在更真实完整的环境中bit-slice优化应当具有更高的效率,同时其抵御测信道攻击的特点意味着更高的安全性。

\subsubsection{多线程结果主要反映了QEMU的局限}
无论是多缓冲区并行加密还是并行解密，在 QEMU-RISC-V 平台上的多线程结果都没有体现出理想扩展性。多数场景下，1线程已经达到最佳或接近最佳结果，线程数增加后反而略有下降。

这一结论更应理解为 QEMU 系统模拟环境的限制，而不是 SM4-CBC 并行策略本身无效。当前结果足以说明算法功能正确、线程逻辑成立，但不足以代表真实多核 RISC-V 硬件上的并行扩展能力。

\subsection{实验局限性}

\begin{enumerate}
    \item \textbf{QEMU 结果只能反映趋势，不能等价替代真实硬件。} 当前所有 RISC-V 结果都建立在系统模拟环境之上，因此更适合比较优化前后相对趋势，而不适合直接作为最终绝对性能指标。
    \item \textbf{bit-slice 的实现价值偏安全性，不偏效率。} 这一点在本次结果中已经得到了明确体现。
    \item \textbf{RISC-V 专用扩展的潜力尚未完全释放。} Zksed 已经完成实现，但其优势在模拟环境中被弱化；更进一步的向量扩展或真实硬件验证仍然是后续可继续推进的方向。
\end{enumerate}

\subsection{结论}

本实验的核心结论可以概括为三点：
\begin{itemize}
    \item [1.]面向 RISC-V 平台时，查表优化与编译器优化是当前最有效、最稳定的性能提升手段；
    \item [2.]Zksed 指令优化已经证明了专用密码指令方向的可行性，但其优势仍需更真实的运行环境才能充分体现；
    \item [3.]bit-slice 在本实验环境中虽然性能不理想，但为常量时间实现和后续面向更强硬件支持的优化提供了实现基础。
\end{itemize}
