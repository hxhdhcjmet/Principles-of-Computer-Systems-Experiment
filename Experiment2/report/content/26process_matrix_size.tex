\subsection{矩阵规模对 Cache 命中率的影响}

为了分析程序工作集大小对 Cache 命中率的影响，本实验进一步改变矩阵乘法程序中的矩阵规模，分别测试 $N=32$、$N=64$、$N=128$ 三种情况下的 Cache 命中率变化。

矩阵规模会直接影响程序运行时的访存工作集大小。当矩阵规模较小时，主要数据可以较容易地被 Cache 容纳；当矩阵规模增大后，程序访问的数据总量迅速增加，Cache 容量不足时会产生更多容量缺失。

\subsubsection{实验设计}

本组实验固定 Cache 配置，改变矩阵规模。固定参数如下：

\begin{itemize}
    \item Cache 块大小：64B；
    \item 映射方式：4 路组相联；
    \item 替换算法：LRU。
\end{itemize}

变化参数为矩阵规模：

\begin{itemize}
    \item $N=32$；
    \item $N=64$；
    \item $N=128$。
\end{itemize}

对于 int 型矩阵而言，每个元素占 4B。因此，单个 $N \times N$ 矩阵大小为：

\[
MatrixSize = N \times N \times 4B
\]

三个矩阵 $A$、$B$、$C$ 的总数据量为：

\[
WorkingSet = 3 \times N \times N \times 4B
\]

\subsubsection{实验现象与分析}

当矩阵规模为 $N=32$ 时，单个矩阵大小为 4KB，三个矩阵总大小为 12KB。在 Cache 容量为 32KB 或 64KB 时，主要数据集可以较好地被 Cache 容纳，因此命中率通常较高。此时 Cache Miss 主要来自首次访问数据时产生的冷启动缺失。

当矩阵规模增大到 $N=64$ 时，单个矩阵大小为 16KB，三个矩阵总大小为 48KB。此时，如果 Cache 容量为 32KB，已经无法完整容纳三个矩阵的数据，容量缺失开始增加；如果 Cache 容量为 64KB，则能够覆盖更大比例的活跃数据，因此命中率会明显优于 32KB 的情况。

当矩阵规模进一步增大到 $N=128$ 时，单个矩阵大小为 64KB，三个矩阵总大小达到 192KB。此时即使 Cache 容量为 64KB，也只能容纳一个矩阵大小的数据，无法完整容纳全部工作集。因此，在普通矩阵乘法实现中，尤其是访问矩阵 $B[k][j]$ 时，由于按列访问导致访存步长较大，Cache Miss 数会显著增加。

\subsubsection{实验总结}

通过不同矩阵规模的对比可以看出，矩阵规模直接决定了程序的工作集大小。当工作集能够被 Cache 较好容纳时，程序命中率较高；当工作集超过 Cache 容量后，容量缺失显著增加，命中率下降。

因此，Cache 性能不仅取决于 Cache 自身参数，也取决于程序数据规模与 Cache 容量之间的匹配关系。对于矩阵乘法这类数据密集型程序，随着矩阵规模增大，若不改变代码访存方式，仅依靠固定容量的 Cache 难以持续保持较高命中率。
