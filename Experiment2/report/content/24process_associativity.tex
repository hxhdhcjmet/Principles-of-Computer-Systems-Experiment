\subsection{相联度对 Cache 命中率的影响}

为了进一步分析 Cache 映射方式对命中率的影响，本实验在固定 Cache 容量、块大小和替换算法的条件下，改变 Cache 的相联度，比较直接映射、组相联映射和全相联映射在同一访存序列下的 Hit/Miss 情况。

本组实验采用矩阵规模为 $N=128$ 时生成的 trace.log 作为输入，固定 Cache 块大小为 64B，替换算法为 LRU，并分别测试不同相联度下的 Cache 命中率。

\subsubsection{实验设计}

本实验固定如下参数：

\begin{itemize}
    \item Cache 大小：64KB； %Cache大小为32KB时1、2、4、8路组相联的hit rate都很小(49%左右)，full时上了95,此时不太好说明影响(如果需要可以加上)
    \item 矩阵规模:$N=128$;
    \item Cache 块大小:64B;
    \item 替换算法:LRU;
    \item 输入文件：$N=128$ 时由 NEMU 生成的 trace.log。
\end{itemize}

改变的参数为 Cache 相联度，包括：

\begin{itemize}
    \item 1 路组相联，即直接映射；
    \item 2 路组相联；
    \item 4 路组相联；
    \item 8 路组相联；
    \item 全相联映射。
\end{itemize}

其中，直接映射可以看作相联度为 1 的特殊组相联 Cache；全相联映射可以看作只有 1 个 Set、所有 Cache Line 都位于同一组内的特殊情况。

\subsubsection{实验现象与分析}

从实验结果可以看出，在固定 Cache 大小为 64KB、块大小为 64B、替换算法为 LRU 的条件下，Cache 命中率并没有随着相联度的增大而单调上升。具体而言，2 路组相联的命中率最高，为 98.16\%，Miss 数为 78164；直接映射的命中率为 97.83\%，略低于 2 路组相联；而 4 路组相联、8 路组相联和全相联的命中率分别下降到 97.26\%、95.46\% 和 96.79\%。

这一现象说明，在本实验的访存序列和模拟器实现条件下，影响命中率的不仅是相联度本身，还包括 Cache 组数、地址分布、替换行为以及程序访存模式等因素。

理论上，提高相联度可以增加同一个 Set 内可容纳的 Cache Line 数量，从而缓解多个主存块映射到同一位置时产生的冲突缺失。但是在 Cache 总容量和块大小固定的情况下，提高相联度会减少 Cache 的组数。例如，当 Cache 容量固定为 64KB、块大小固定为 64B 时，1 路、2 路、4 路和 8 路对应的组数依次减少。组数减少后，更多主存块会被映射到同一个 Set 中，使得该 Set 内部的替换压力增加。

因此，相联度提高带来的“每组可容纳更多块”的好处，可能会被“组数减少导致更多地址集中到同一组”的影响部分抵消。对于本实验中的矩阵乘法 trace，2 路组相联在冲突缓解和组数保留之间取得了较好的平衡，因此命中率最高；而 8 路组相联虽然每组容量更大，但组数更少，局部 Set 的竞争更加集中，导致 Miss 数反而增加。

全相联映射理论上可以消除组索引冲突，因为任意主存块都可以放入任意 Cache Line 中。但在实际模拟时，全相联也意味着所有块都在同一个集合中参与替换，替换算法对整体访问序列更加敏感。因此，本实验中全相联的命中率虽然高于 8 路组相联，但并没有超过 2 路组相联。

\subsubsection{实验总结}

通过本组实验可以看出，相联度对 Cache 命中率的影响不是简单的单调关系。理论上，提高相联度有助于减少冲突缺失；但在 Cache 总容量固定时，相联度提高会同时减少 Set 数量，使更多地址竞争同一组中的 Cache Line。因此，相联度过高不一定带来更高命中率。

本实验数据表明，2 路组相联在当前矩阵乘法 trace 下取得了最高命中率，说明适中的相联度能够在冲突缺失和组数规模之间取得较好平衡。实际 Cache 设计也需要在命中率、硬件复杂度、访问延迟和功耗之间折中，而不是简单追求更高的相联度。
